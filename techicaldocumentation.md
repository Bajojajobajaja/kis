ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ)
на разработку корпоративной информационной системы (КИС) ООО «Nexus» в виде веб-приложения

Общие сведения
1.1. Наименование: «КИС Nexus: Продажи/CRM — Сервис/Ремонт — Склад/Запасы — Финансы/Отчётность».
1.2. Заказчик: ООО «Nexus» (автодилерский центр).
1.3. Назначение: сквозная автоматизация ключевых процессов: обработка обращений, продажи авто, сервисное обслуживание, учёт запасов, финучёт и отчётность.

Цели и ожидаемые результаты
2.1. Цели:
— единая информационная среда для отделов продаж, сервиса, логистики/склада, финансов;
— сокращение ручного ввода и устранение дублирования данных;
— повышение прозрачности статусов сделок/ремонтов/остатков и ускорение отчётности.
2.2. Ожидаемые эффекты (целевые KPI допустимо закрепить отдельным приложением):
— сокращение времени обработки заявок;
— снижение ошибок в договорах/счётах/актах;
— оптимизация складских запасов;
— ускорение подготовки финансовой/управленческой отчётности.

Границы системы и пользователи
3.1. Роли (минимальный набор):
— Генеральный директор (просмотр KPI, отчёты, контроль);
— Руководитель продаж / Старший менеджер;
— Менеджер по продажам;
— Администратор заказов (входящие заявки/документооборот);
— Начальник сервиса;
— Мастер-приёмщик;
— Диагност/Механик (исполнители работ);
— Начальник логистики / Логист;
— Кладовщик;
— Главный бухгалтер / Бухгалтер / Экономист;
— Маркетолог (опционально, если потребуется коммуникационный модуль).
3.2. Каналы ввода заявок: вручную оператором + интеграции (сайт/телефон/мессенджеры) — поэтапно.

Архитектура и общие технические требования
4.1. Архитектурный стиль: микросервисы, взаимодействие по REST/gRPC + событийная шина.
4.2. Язык: Golang для backend-сервисов.
4.3. Веб-интерфейс: SPA (любая современная связка), единая точка входа.
4.4. Базовые инфраструктурные компоненты:
— API Gateway (маршрутизация, rate limit, auth);
— Identity & Access (RBAC);
— Audit/Logging (журнал действий);
— Notification (уведомления);
— Reporting/BI (витрины/дашборды);
— Event Bus (Kafka/RabbitMQ/NATS — выбирается проектом);
— Observability (метрики/трейсинг).
4.5. Данные:
— каждая подсистема владеет своими данными (подход database-per-service);
— межсервисные связи через идентификаторы и события;
— единые справочники/классификаторы (авто, запчасти, услуги, контрагенты, сотрудники, счета/статьи) как отдельные сервисы или как bounded context в «master data».
4.6. Безопасность:
— RBAC на уровне эндпоинтов и объектов (например, менеджер видит “своих” лидов/сделки);
— аудит действий (создание/изменение документов, финансовые операции, списания);
— резервное копирование БД;
— TLS, хранение секретов в vault/secret manager.
4.7. Нефункциональные требования (минимум):
— доступность: 99.5% (на этапе MVP можно ниже, фиксируется договором);
— производительность: основные операции (поиск клиента/авто/VIN, открытие карточки сделки/ремонта) до 2 сек при типовой нагрузке;
— целостность: финансовые операции и складские движения — транзакционно в рамках сервиса, межсервисно — через саги/компенсации.

Общая доменная модель (ключевые сущности)
— Клиент (физ/юр лицо), Контактные данные, История взаимодействий.
— Лид/Обращение, Задачи/активности.
— Сделка (продажа авто), Предложение, Договор, Счёт, Оплата, Передача авто.
— Автомобиль (VIN, марка/модель, новый/б/у, статус: в наличии/в резерве/продан/в сервисе).
— Заказ-наряд, Работы, Нормо-часы, Диагностика, Запчасти/расходники, Акт выполненных работ.
— Номенклатура (запчасти), Остатки, Партии, Движения, Инвентаризация, Заказ поставщику, Приход/накладные.
— Финансовые проводки/операции, Статьи, Отчёты (P&L, ДДС, баланс — по уровню учёта).

Подсистемы и микросервисы

6.1. Подсистема управления продажами и CRM
Назначение: приём/обработка обращений, ведение клиентской базы, сделки по продаже авто, документооборот продажи.

Рекомендуемое разбиение на микросервисы:
A) crm-contacts
Функции:
— CRUD клиентов/контрагентов, контактов, предпочтений;
— история взаимодействий (звонок/письмо/встреча), привязка к лидам/сделкам.
Данные: client_id, контрагент, контактные лица, теги/сегменты.
API: /clients, /contacts, /interactions.
События: ClientCreated/Updated.

B) crm-leads
Функции:
— регистрация лидов из каналов;
— статусы (новый/в работе/квалифицирован/закрыт), назначение ответственного;
— SLA/дедлайны, задачи менеджерам.
API: /leads, /activities.
События: LeadCreated, LeadAssigned, LeadStatusChanged.

C) sales-deals
Функции:
— воронка продаж (сделка, этапы, резервирование авто);
— расчёт цены: базовая + опции/скидки + кредит/лизинг (если внедряется);
— контроль статуса “оплата/договор/передача”.
API: /deals, /deals/{id}/stages, /deals/{id}/reserve-vehicle.
События: DealCreated, DealWon/Lost, VehicleReserved, VehicleSold.

D) sales-documents
Функции:
— генерация договора купли-продажи, счёта, акта приёма-передачи, чека (как документ);
— шаблоны документов, версия шаблонов.
API: /documents/templates, /documents/generate.
События: DocumentGenerated.

E) pricing (опционально на MVP можно внутри sales-deals)
Функции:
— правила скидок, прайсы, акции, комиссионные;
— расчёт итоговой стоимости.
API: /pricing/calc.

Интеграции:
— с inventory: проверка наличия авто, резервирование VIN/единицы;
— с finance: выставление счёта, фиксация оплаты;
— с notification: подтверждения клиенту (SMS/email).

Ключевые сценарии:
— «входящее обращение → лид → квалификация → подбор авто → сделка → договор/счёт → оплата → передача → списание авто со склада».

6.2. Подсистема сервисного центра и ремонта
Назначение: запись в сервис, заказ-наряды, диагностика, планирование работ, учёт запчастей в ремонте, расчёт стоимости, закрывающие документы.

Микросервисы:
A) service-appointments
Функции:
— календарь записи, слоты, загрузка постов/мастеров;
— создание визита: клиент + авто + причина.
API: /appointments, /slots.
События: AppointmentCreated/Rescheduled/Cancelled.

B) service-workorders
Функции:
— заказ-наряд (WO): статусы (принят/в диагностике/в работе/ожидает запчасти/готов/выдан);
— назначение исполнителей, контроль сроков.
API: /workorders, /workorders/{id}/status.
События: WorkorderCreated, WorkStarted, WorkPaused, WorkCompleted.

C) service-diagnostics
Функции:
— результаты диагностики, перечень неисправностей;
— рекомендации работ/запчастей.
API: /diagnostics.
События: DiagnosticCompleted.

D) service-labor-catalog
Функции:
— справочник работ/услуг, нормы времени, тарифы;
— расчёт стоимости работ по нормо-часам.
API: /labor, /labor/{id}.
События: LaborUpdated.

E) service-parts-usage
Функции:
— списание/резерв запчастей под заказ-наряд;
— контроль дефицита: автозаказ/заявка логисту.
API: /workorders/{id}/parts.
События: PartsReserved, PartsConsumed, PartsShortageDetected.

F) service-billing
Функции:
— смета/счёт по заказ-наряду (работы + запчасти);
— акт выполненных работ, закрытие.
API: /service-invoices, /acts.
События: ServiceInvoiceIssued, WorkorderClosed.

Интеграции:
— с inventory: резерв/списание запчастей;
— с finance: проводки и оплаты по ремонту;
— с CRM: история клиента, уведомления.

Ключевые сценарии:
— «запись → приёмка → диагностика → согласование работ → резерв/заказ запчастей → ремонт → счёт/акт → оплата → выдача авто».

6.3. Подсистема управления запасами и складом
Назначение: учёт автомобилей и запчастей, движения, закупки, инвентаризация, минимальные остатки.

Микросервисы:
A) masterdata-catalog
Функции:
— справочники: авто-модели, номенклатура запчастей, единицы измерения, совместимость;
— единые коды/артикулы.
API: /catalog/cars, /catalog/parts.
События: CatalogItemUpdated.

B) inventory-stock
Функции:
— остатки по складам/ячейкам, партии, серийники (VIN как серийник);
— операции: приход/перемещение/резерв/списание.
API: /stock, /stock/movements, /stock/reserve.
События: StockChanged, StockReserved, StockReleased.

C) inventory-procurement
Функции:
— заявки на закупку, заказы поставщикам, статусы поставок;
— расчёт потребности: min/max, точка заказа, сезонность (опционально).
API: /procurement/requests, /purchase-orders.
События: PurchaseOrderCreated, DeliveryReceived.

D) inventory-receiving
Функции:
— приёмка по накладным, сопоставление с заказом;
— контроль расхождений, возвраты.
API: /receipts, /receipts/{id}/discrepancy.

E) inventory-audit
Функции:
— инвентаризация: задания, ведомости, расхождения, корректировки остатков;
— журнал инвентаризаций.
API: /inventory-checks.

Интеграции:
— продажи: списание/перевод авто в “продано”;
— сервис: резерв/списание запчастей под WO;
— финансы: себестоимость, приходные документы, оценки остатков.

Ключевые сценарии:
— «поступление → размещение → резерв (продажа/ремонт) → списание → инвентаризация → корректировка».

6.4. Подсистема финансового учёта и отчётности
Назначение: фиксация оплат/начислений, управленческий и (по возможности) бухгалтерский контур, отчётность и аналитика.

Микросервисы:
A) finance-ledger
Функции:
— единый реестр финансовых операций: начисления, оплаты, возвраты;
— статьи доходов/расходов, контрагенты, центры ответственности.
API: /ledger/entries, /ledger/post.
События: LedgerEntryPosted.

B) finance-invoicing
Функции:
— счета по продажам и сервису (если не делать в домене продаж/сервиса);
— статусы оплаты, сверки.
API: /invoices, /payments.

C) finance-costing
Функции:
— себестоимость: авто, запчасти, работ;
— маржинальность сделок/заказ-нарядов.
API: /costing/calc.

D) finance-reporting
Функции:
— отчёты: доходы/расходы (P&L), ДДС, валовая прибыль по направлениям (продажи/сервис), оборачиваемость запасов;
— выгрузки (Excel/PDF), расписание отчётов.
API: /reports, /reports/export.
События: ReportGenerated.

E) analytics-marts (рекомендуется как отдельный контур)
Функции:
— витрины/агрегации для дашбордов, near real-time;
— приём событий из шины, денормализация.
Хранилище: отдельная БД/колонночное хранилище (по возможностям проекта).

Интеграции:
— продажи/сервис: выставления и оплаты;
— склад: приход/списание → себестоимость → отчёты.

Межсервисные контракты: события и интеграции (минимальный перечень)
События (примерный каркас):
— crm: ClientCreated/Updated, LeadCreated, DealWon;
— sales: VehicleReserved, VehicleSold, InvoiceIssued;
— service: WorkorderCreated, PartsConsumed, WorkorderClosed;
— inventory: StockReserved/Released, StockChanged, DeliveryReceived;
— finance: PaymentReceived, LedgerEntryPosted.

Правила согласованности:
— междоменные операции (продажа авто, закрытие заказ-наряда) реализуются как сага:

домен инициатор публикует событие,

инвентори резервирует/списывает,

финансы фиксируют начисление/оплату,

при сбоях — компенсирующие действия (снятие резерва, сторно).

Требования к интерфейсу (общие)
— единый веб-кабинет с меню по подсистемам;
— быстрый глобальный поиск: клиент/телефон, VIN, номер заказ-наряда, номер сделки/счёта;
— карточки объектов (клиент, авто, сделка, WO, позиция склада) с таймлайном событий;
— контрольные панели руководителей: продажи/сервис/склад/финансы.

Требования к журналированию и доступам
— аудит: кто/когда/что изменил (до/после) для критичных сущностей: договоры, счета, оплаты, списания, корректировки склада;
— матрица прав (RBAC) минимум по подсистемам + по операциям (create/update/approve/export).

Тестирование и приёмка
— модульные тесты (core-логика расчётов, статусы, переходы);
— интеграционные тесты (саги: продажа, закрытие ремонта);
— нагрузочные тесты (поиск, списания, генерация отчётов);
— критерии приёмки: выполнение ключевых сценариев end-to-end + отчёты + аудит.
